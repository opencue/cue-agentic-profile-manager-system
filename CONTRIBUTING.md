# Contributing to soul

soul is the user's personal Claude Code / Codex skill library plus the
profile system that bundles skills, MCPs, and env for specific kinds of work.
This document covers the three contribution surfaces.

## 1 — Adding a profile

A profile lives at `profiles/<name>/profile.yaml` and the directory name must
equal the `name:` field.

```bash
soul new my-profile               # empty template
soul new my-profile --from-scan   # auto-bucket installed skills
soul new my-profile --seed core   # copy an existing profile
```

The schema is documented in [`profiles/SCHEMA.md`](./profiles/SCHEMA.md);
validation in [`docs/profiles/auto-generation.md`](./docs/profiles/auto-generation.md).

Before opening a PR:

```bash
soul validate my-profile          # schema + lint
soul use my-profile --here        # smoke test in CWD
```

Keep the profile small. The point of profiles is token discipline — a profile
with more than ~25 skills is almost certainly two profiles wearing a trench
coat.

## 2 — Adding a local skill

Local skills live under `soul/skills/skills/<category>/<skill-name>/SKILL.md`.
They're plain Claude Code / Codex skills — the per-skill anatomy is in
[`skills/README.md`](./skills/README.md) § Contributing.

A new skill is picked up by `soul scan` automatically once the SKILL.md file
exists. To get it into a profile:

```yaml
# profiles/<existing>/profile.yaml
skills:
  local:
    - <category>/<skill-name>     # the path under soul/skills/skills/
```

## 3 — Adding an MCP server

MCP servers are registered in `soul/mcps/configs/claude.sanitized.json` and
`soul/mcps/configs/codex.sanitized.json`. The snapshot is regenerated by
`recodeee/mcps`'s tooling — see [`mcps/README.md`](./mcps/README.md) for the
canonical workflow.

Once an MCP id is in the sanitized config, a profile can reference it:

```yaml
# profiles/<name>/profile.yaml
mcps:
  - my-mcp-id
```

## PR workflow (for dev-mode contributors)

If you are running soul in dev mode with the parallel-agents tier (Colony +
`gx`), the contract is:

- **No edits on the primary checkout.** `gx branch start <branch>` first, then
  `cd` into the printed worktree path.
- **Claim files before editing** via `mcp__colony__task_claim_file` (or
  `gx locks claim --branch <br> <files...>`).
- **Finish through `gx branch finish --branch <br> --via-pr --wait-for-merge
  --cleanup`.** Never raw `git push`/`merge`.

If you're a single-operator contributor (no Colony, no fleet), just open a
feature branch and PR normally. The gx/Colony rules only bind agents running
inside a multi-agent context.
